## 简介

Markdown 是一种轻量级标记语言，由 John Gruber 于 2004 年创建。它允许人们使用易读易写的纯文本格式编写文档，然后转换成有效的 HTML 文档。

**主要特点：**
- **语法简单直观**：标记符（如 `#`, `*`）来源于电子邮件和纯文本的书写习惯，易于记忆。
- **兼容 HTML**：可以直接在 Markdown 中插入 HTML 标签，用于实现更复杂的样式。
- **可被多种工具解析**：有大量编辑器、平台和转换工具支持 Markdown。
- **支持多种输出格式**：可轻松转换为 HTML、PDF、Word、Epub 等格式。

**基本理念：**
1.  易于阅读和编写：即使是未经渲染的原始 Markdown 文档，也应该具备良好的可读性。
2.  可以转换为有效的 XHTML（或 HTML）。
3.  语法完全由标点符号组成，借鉴了网络文本格式的既有习惯。

**常见用途：**
- 编写项目的 `README` 文档（如 GitHub）。
- 在论坛、博客平台撰写文章。
- 撰写可快速转换为演示文稿或报告的技术笔记。
- 作为笔记软件（如 Obsidian、Notion）的底层格式。

## 前言

### 水平线

水平线用于在文档中创建视觉上的分隔，通常表现为一条横贯页面的横线。

#### 语法
在一行中使用三个或更多的星号 `*`、减号 `-` 或下划线 `_` 即可生成水平线。符号之间可以插入空格。

## 标题

Markdown 支持两种标题语法：类 Setext 和类 atx 形式。现代编辑器和平台普遍推荐使用 **类 atx 形式**。

### 类 Setext 形式 (下划线形式)
使用等号 (`=`) 表示一级标题，连字符 (`-`) 表示二级标题。符号的长度至少与标题文本一样长。


一级标题
----
二级标题
====


### 类 atx 形式 (井号形式) - 推荐
在行首插入 1 到 6 个 `#` 字符，对应 HTML 中的 `<h1>` 到 `<h6>` 标题级别。

# 一级标题
## 二级标题
### 三级标题
#### 四级标题
##### 五级标题
###### 六级标题



**最佳实践与注意事项：**
1.  **空格**：在 `#` 和标题文字之间**强烈建议**加一个空格（如 `# 标题`），这是 CommonMark 等现代标准的要求，能保证最好的兼容性。
2.  **可选闭合符号**：可以在标题行的末尾也加上 `#`，数量不必与开头相同，这纯粹是装饰性的（如 `### 标题 ###`）。大多数情况下无需使用。
3.  **前后空行**：标题的上一行和下一行最好是空行，这能确保 Markdown 解析器正确识别标题，避免格式混乱。
4.  **避免断开**：标题行内不应包含任何换行符。

## 段落与换行

理解段落和换行的区别是编写格式清晰文档的基础。

### 段落
一个段落由一个或多个连续的文本行组成。Markdown 通过**空行**来区分不同的段落。当渲染时，段落通常会被包裹在 `<p>` 标签中。

这是第一段落。这是一段很长的文字，  
在源码中可以折行书写，但只要中间没有空行，  
渲染时会显示为同一个段落。

这是第二个段落。因为它和上一段之间有一个空行。


**注意**：在源码中为了让一行不至于太长而手动换行，并不会在渲染结果中产生新段落。( 这个不一定，有些工具会自动处理 例如 vscode )

### 换行 (强制换行)
如果需要在段落内进行强制换行（即生成 `<br>` 标签），有两种标准方法：

1.  **两个空格后回车**：在行尾先输入两个空格，再按回车键。这是最通用的 Markdown 标准语法。
    这是第一行  
    这是第二行，它与第一行属于同一段落但强制换行。
2.  **使用 HTML 的 `<br>` 标签**：直接插入 `<br>`，兼容性最好。
    这是第一行<br>
    这是第二行。

**核心区别总结：**
- **空行分隔 → 新段落** (`<p>`)
- **两空格后回车 或 `<br>` → 同一段落内换行** (`<br>`)

## 文本样式

Markdown 提供了简单的语法来为文本添加基础的强调效果。

### 强调 (粗体与斜体)
使用 `*` 或 `_` 作为标记符号。为了清晰和避免歧义，通常建议：用 `**` 表示粗体，用 `*` 表示斜体。

***这是一段粗斜体文字，通过星号和双星号***
**这是一段粗体文字，通过双星号**
*这是一段斜体文字，通过星号*

__这是一段粗体文字，通过下划线__
_这是一段斜体文字，通过下划线_

嵌入的效果：**这是粗体** 和 *这是斜体* 以及 这是 ***粗斜体***。


**注意**：为了可读性和兼容性，在需要强调的词语或短语**内部**不要有空格，例如应写 `**重要**` 而非 `** 重要 **`。

### 删除线
使用两个波浪符 `~~` 包裹文本。这是 GFM (GitHub Flavored Markdown) 等扩展标准的一部分，被广泛支持。

~~这是一段被删除的文字~~
**注意**：删除线通常用于表示内容不再有效或需要移除。


### 其他样式 (非原生支持)
以下功能并非所有 Markdown 解析器都支持，常见于 Typora、Obsidian 或某些特定平台（如 GitLab）。

- **高亮**：使用两个等号 `==` 包裹文本。
请特别注意 ==这部分关键内容==。

- **下标**：使用波浪符 `~` 包裹。
水的化学式是 H~2~O。
- **上标**：使用脱字符 `^` 包裹。
勾股定理：a^2^ + b^2^ = c^2^。

**提示**：对于下标、上标等复杂格式，最可靠的方法是直接使用对应的 HTML 标签：`<sub>下标</sub>` 和 `<sup>上标</sup>`。

## 列表

列表用于呈现条目化的信息，分为有序列表和无序列表。

### 无序列表
使用星号 `*`、加号 `+` 或减号 `-` 作为列表标记，后跟一个空格。**建议在整个文档中保持标记符号的一致性**。
- 无序列表项 1
- 无序列表项 2

+ 无序列表项 1
+ 无序列表项 2

- 无序列表项 1
- 无序列表项 2

### 有序列表
使用数字加英文句点 `.` 作为标记，后跟一个空格。**关键点**：列表项的序号并不影响渲染结果，最终总会按顺序显示。
1. 穿裤子
2. 穿袜子
3. 穿鞋子
4. 穿帽子

有序列表的顺序只能从 1 开始，不能从其他数字开始，也不能跳过数字。但是可以设置开头的数字，后续的数字谁自动从开头数字开始递增 例如：

5. 穿裤子
2. 穿袜子
3. 穿鞋子
4. 穿帽子

### 嵌套列表
支持嵌套列表，只需在子列表项前增加正确的缩进。

1. 吃饭
   - 吃菜
        1.先把碗递过去
        2.把菜夹到碗里
   - 喝汤
2. 开车
   1. 打火
        - 脚踩刹车和离合，插入钥匙
        - 转动钥匙，启动
   2. 加油门，走

**注意**：嵌套列表需要使用正确的缩进（通常是两个空格），以确保解析器正确识别。


### 任务列表 (复选框)
这是 GFM 的扩展语法，被 GitHub、GitLab、许多笔记软件等广泛支持。用 `- [ ]` 表示未完成，`- [x]` 表示已完成（`x` 不区分大小写）。
- [ ] 吃饭
- [x] 睡觉
  - [ ] 垫枕头
  - [ ] 盖被子

### 定义列表
部分解析器（如 PHP Markdown Extra、某些静态网站生成器）支持。用于创建术语及其解释。
- 调制解调器
  : 调制解调器是一种将数字信号转换为模拟信号或反之的设备。
- 基带芯片
  : 基带芯片是一种用于处理基带信号的集成电路。

**通用建议**：在列表项前后，尤其是列表与其他内容之间，保留空行，以确保解析正确。

## 链接

Markdown 支持多种方式插入超链接。

### 1. 行内链接 (Inline Link) - 最常用
语法格式：[ **链接文本** ] ( **链接地址** **"可选标题"** )
- **链接文本**：显示在页面上的可点击文字。
- **链接地址**：可以是 URL（`https://...`）或本地文件路径（`./docs/file.md`）。
- **可选标题**：当鼠标悬停在链接上时显示的提示文字，用英文引号包裹。

这是我的[GitHub 主页](https://github.com/your-username "点击访问")。
[ 敢杀我的马?! ](https://www.bilibili.com/video/BV1yt4y1Q7SS?spm_id_from=333.788.player.switch&vd_source=dbfefd0cc37b2895be48a923d6db2d8d "敢杀我的马?!" )

### 2. 参考式链接 (Reference Link) - 适合多次引用
将链接地址的定义和引用分开，使文档更整洁。特别适合同一链接在文中多次出现时使用。
* 语法格式-引用的地方：   [ **链接文本** ] [ **标识符** ]
* 语法格式-定义的地方：   [ **标识符** ]: **链接地址**  **"可选标题"**

这个是我的[GitHub主页][ GitMainPage ]。

[ GitMainPage ]: https://github.com/your-username "点击访问"

你甚至可以省略标识符直接使用链接文本作为标识符

这个是我的[GitHub主页]

[ GitHub主页 ]: https://github.com/your-username "点击访问"

**注意:** 链接统一放在文章末尾，方便管理。


### 3. 自动链接
用尖括号 `< >` 将 URL 或邮箱地址包裹起来，会自动转换为可点击的链接。
- 网址： <https://github.com/your-username>
- 邮箱： <your-email@example.com>
- **不包好像也行:**  your-email@example.com


### 4. 锚点链接 (链接到文档内部)
链接到同一文档内的某个标题。语法是 `#` 加上标题名称（需转为小写，空格用 `-` 连接）。

[目录的跳转就是这么实现的，点我查看目录](#目录)
[标题名带空格的时候这么写,把空格用-代替](#html-支持)

**注意**：生成的实际锚点 ID 可能因解析器而异，简单标题通常有效。

### 最佳实践
- **保持描述性**：链接文本应说明去向，避免使用“点击这里”。
- **引用式管理**：长文档或链接众多时，使用参考式链接将所有链接地址统一放在文档末尾，便于管理和更新。

## 图片

插入图片的语法与链接非常相似，只是在开头多了一个感叹号 `!`。

### 基本语法

![ **图片描述** ] ( **图片地址** **"可选标题"** )

- **图片描述**：这是**必填项**。当图片无法加载时，会显示这段文本。它对无障碍访问（屏幕阅读器）也至关重要。
- **图片地址**：图片的地址。可以是：
    - **绝对路径**：`https://example.com/image.png`
    - **相对路径**：`./images/photo.jpg` (推荐项目内使用)
- **"可选标题"**：鼠标悬停在图片上时显示的文字提示。

**示例**：

![ github emoji ]( https://github.githubassets.com/images/icons/emoji/octocat.png "img")

### 参考式图片
与参考式链接类似，将图片地址定义在文档其他位置，使行内代码更简洁。**但请注意，此语法并非所有解析器都完美支持。**

![ 参式链接图片 ][ PicEmoji ]

[ PicEmoji ]: https://github.githubassets.com/images/icons/emoji/octocat.png "img"


### 带链接的图片
如果想为图片添加超链接，将图片语法整个包裹在链接语法中即可。


**示例**：点击图片跳转到GitHub。
[ ![ github emoji ]( https://github.githubassets.com/images/icons/emoji/octocat.png "img")]( https://github.com "点击访问")


### 设置图片尺寸
标准的Markdown语法**不支持**直接指定宽高。有以下两种常用解决方案：

1.  **使用HTML `<img>` 标签**（最可靠、兼容性最好）：

```html
<img src="https://github.githubassets.com/images/icons/emoji/octocat.png" alt="描述" width="200" height="200">
```

<img src="https://github.githubassets.com/images/icons/emoji/octocat.png" alt="描述" width="200" height="200">


2.  **部分平台/扩展支持的自定义语法**（如某些Markdown编辑器或静态网站生成器）：
    ```
    ![描述](image.jpg){: width="200" height="100"}
    ```
    *这种语法并非通用标准，需确认你的工具是否支持。*
    ![描述][ PicEmoji ]{: width="200" height="200"}

### 最佳实践
1.  **务必填写有意义的替代文本**，这是良好的写作习惯和无障碍要求。
2.  对于项目文档，尽量使用**相对路径**引用本地图片，便于项目迁移和版本管理。
3.  考虑性能，大尺寸图片应进行压缩。

## 代码

Markdown 可以清晰地展示行内代码和代码块，这是其作为技术文档工具的核心优势之一。

### 行内代码 (Inline Code)
使用单个反引号 `` ` `` 将一小段代码或关键字包裹起来。这通常用于在段落中提及函数名、变量、命令等。

例如：
在git中 `git commit` 是提交代码的命令。
请运行 `npm install ` 命令安装依赖。


**注意**：如果要在行内代码中显示反引号本身，可以用双反引号包裹：`` `` `反引号` `` `` 会渲染为 `` `反引号` ``。

### 代码块 (Code Block)
用于展示多行代码。有两种主流语法：

1.  **围栏代码块 (Fenced Code Blocks) - 强烈推荐**
    使用三个反引号 ` ``` ` 或三个波浪线 `~~~` 包裹代码，并在起始围栏后**指定语言**以实现语法高亮。

    
    ```python
    def greet(name):
        print(f"Hello, {name}!")
    
    greet("World")
    ```
    
    ~~~javascript
    console.log("Hello, World!");
    ~~~
    

2.  **缩进代码块 (Indented Code Blocks)**
    在每一行代码前加一个制表符（Tab）或**四个空格**。此方法无法指定语言，通常不进行语法高亮，且容易与列表嵌套混淆，现已较少使用。

    这是一个用四个空格缩进的代码块。
    function oldSchool() {
        return true;
    }

### 语法高亮
在围栏代码块的起始反引号后指定语言标识符，即可激活语法高亮。例如：
- `bash` 或 `sh`
- `c`, `cpp`, `java`, `python`, `javascript`, `html`, `css`, `json`, `yaml`, `sql`

### 显示代码块中的反引号
如果代码块内容本身包含三个连续的反引号，需要用更多层的反引号来包裹。例如，要展示一个用反引号标记的代码块，可以这样写：

````markdown
    ``` python 
    def greet(name):
        print(f"Hello, {name}!")
    ```
````    
上面最外层的四个反引号是“围栏”，里面包裹的内容（包括三个反引号）会被原样显示为代码。

### 行号显示：
有些解析器或扩展支持在围栏后添加属性来显示行号。

```python {.line-numbers}
# 这可能会显示行号1
# 这可能会显示行号2
# 这可能会显示行号3
# 这可能会显示行号4
```

### 高亮指定行：
高亮显示代码块中的特定行。

```python hl_lines="2 4-5"
# 第2行，以及第4到第5行可能会被高亮显示
# 这行不会被高亮显示
# 这行会被高亮显示
# 这行也会被高亮显示
# 这行不会被高亮显示 
```

### 最佳实践
1.  **行内代码**：尽量少用，以免影响阅读体验。如果需要展示代码片段，使用代码块。
2.  **代码块**：使用围栏代码块，并指定语言以实现语法高亮，这是最清晰、兼容性最好且功能最丰富的方式。
3.  **行号和行高亮**：根据需要选择是否启用，但要注意不要过度使用，以免影响文档可读性。

## 表格

表格是 GFM (GitHub Flavored Markdown) 等扩展中最实用的功能之一，用于清晰地展示结构化数据。

### 基础语法
表格由管道符 `|` 分隔单元格，由连字符 `-` 分隔表头和表体。

| **表头1** | **表头2** | **表头3** |
| :-------: | :-------: | :-------: |
| 单元格1   | 单元格2   | 单元格3   |
| 单元格4   | 单元格5   | 单元格6   |


**关键点解析：**
1.  **表头分隔线**：第二行（`|---|---|`）是必须的，它定义了表格的列和对齐方式。
2.  **对齐方式**：通过在分隔线的冒号 `:` 来设置：
    - `:---` 或 `:----` 表示 **左对齐** (默认)
    - `:---:` 表示 **居中对齐**
    - `---:` 表示 **右对齐**
3.  **单元格内的管道符**：如果需要在单元格内显示管道符 `|`，请使用其HTML实体 `&#124;`。
4.  **空单元格**：空单元格会被渲染为空格。
5.  **换行**：单元格内不支持换行，但可以通过添加多个空行来模拟换行效果。
6.  **简化写法**：最外层的管道符 `|` 不是必须的，这可以使源码更整洁。
7. 表格单元格内支持有限的 Markdown 语法，通常包括：
    - **粗体**、*斜体*、~~删除线~~
    - `行内代码`
    - 简单的[链接](https://example.com)
    - 不支持在单元格内使用多行元素，如代码块、列表、引用块或嵌套表格。


下面用一个包含上述关键点的表格来展示：


| **表头1** | **表头2** | **表头3** | 表头4
| :-------  | :-------: | -------: | :---: 
| *左对齐*    | **居中对齐**   | ***右对齐***   |
| [百度](www.baidu.com)    | ~~单元格5~~   | 使用 `printf` 打印日志   |
||换不了行的老弟||
||没招啊| ![ 插入图片 ]( https://github.githubassets.com/images/icons/emoji/octocat.png "图片") |


### 最佳实践
1.  **保持简洁**：Markdown 表格不适合处理复杂、跨行跨列的布局，复杂表格建议用 HTML 或截图。
2.  **注意空格**：在管道符和单元格内容间添加空格，可以极大提高源码的可读性。
3.  **预先对齐**：在编辑时使用等宽字体并调整好对齐方式，方便后续维护。
4.  **考虑替代方案**：如果数据量很大或需要排序、筛选，考虑将数据放在代码块中（如 JSON、CSV格式），或提供外部文件链接。

## 引用

引用块用于突出显示引自其他来源的文本、重要的说明或提示。

### 基本语法
在段落或文本行前添加一个右尖括号 `>` 和一个空格即可创建引用块。

>这是一个引用块。它会以特殊的样式呈现，通常有左侧竖线或缩进背景。
这是同一引用块内的第二行。


### 多层嵌套引用
通过叠加 `>` 符号可以创建嵌套引用，常用于邮件或对话式引用。


>这是第一层引用。
>>这是第二层引用。
>>>这是第三层引用。


### 引用块中的其他元素和语法
引用块内可以包含几乎所有的其他**Markdown元素**以及支持**大部分Markdown语法**,包括：
- **粗体**、*斜体*、~~删除线~~
- `行内代码`
- [链接](https://example.com)
- **列表**
- **代码块**
- **图片**

>这是一个引用块，里面可以包含：
> - **列表**
> - `行内代码`
> # 一个代码块
> print("Hello from a quote")
> os.system("echo Hello from a quote")
> cv2.imshow("Hello from a quote")
> ```
> - [链接](https://example.com)
> - **嵌套引用**
> ### 一个标题
> ```python


**重要规则**：当在引用块内插入代码块、列表等需要缩进的元素时，需要在原有的 `> ` 基础上，再加上这些元素本身所需的缩进（通常是4个空格或1个制表符）。

### 连续段落与懒人写法
如果引用内容包含多个段落，有两种写法：

1.  **标准写法**：每个段落前都加 `>`
    ```
    > 这是第一个段落。
    >
    > 这是第二个段落，它们同属于一个引用块。
    ```

2.  **懒人写法**：只在第一行前加 `>`，后续段落通过行首留空来“延续”引用。（并非所有解析器都完美支持）
    ```
    > 这是第一个段落。
      这是延续的文本，在某些解析器中仍属于引用。
    ```

**建议**：对于重要的文档，使用**标准写法**以确保兼容性。

### 实际应用场景
- **引述他人的话或文献**。
- **突出显示注意事项、提示、警告**（常与表情符号结合使用）。
- **在回复或讨论中模拟对话结构**。


**关键规则**：
1.  **符号数量**：至少需要 **三个** 相同的符号。
2.  **独立一行**：该行**只能包含这些符号和空格**，不能有任何其他字符。
3.  **符号间空格**：在符号之间插入空格（如 `- - -`）是完全允许的，且能提高源码的可读性。

### 错误示例与解析

--- 这不是水平线 ---
*这行文字因为包含其他文本，所以不会渲染成水平线，而是一个三级标题（类Setext形式）。*

--
*只有两个符号，不足以构成水平线。*

### 最佳实践
- **选择一种样式**：在整篇文档中，建议统一使用一种符号（如 `---`）来创建水平线，以保持风格一致。
- **增加可读性**：使用带空格的写法（如 `- - -`）能让源码的结构更清晰。
- **明确用途**：水平线是强有力的视觉分隔符，适用于分隔主要章节或完全不同的内容部分。对于较小的内容分组，使用标题通常是更好的选择。

## HTML 支持

Markdown 的核心理念之一是**与 HTML 兼容**。你可以在 Markdown 文档中直接使用 HTML 标签，这为实现复杂格式或 Markdown 本身不支持的功能提供了可能。

### 基本原则
1.  **块级 HTML 元素**：如 `<div>`、`<table>`、`<pre>`、`<p>` 等，必须**单独成块**，前后需有空行，且其内部不应使用 Markdown 语法。
2.  **行内 HTML 元素**：如 `<span>`、`<cite>`、`<del>` 等，可以直接在 Markdown 段落中使用。
3.  **标签内的 Markdown**：HTML 块级元素**内部**的 Markdown 语法将**不会被解析**。而行内元素内部则取决于具体的解析器。

### 常用场景与示例

#### 1. 嵌入更复杂的样式
当 Markdown 的基本样式（如颜色、大小）无法满足需求时，可以使用 HTML 标签配合行内样式。

这是一个 <span style="color: red; font-weight: bold;">红色加粗</span> 的文本。

<div style="background: #f0f0f0; padding: 1em; border-left: 4px solid #ccc;"> 这是一个有背景色和左边框的提示框。 此 div 块内的 **Markdown 语法不会被解析**。 </div> ```

#### 2. 插入 Markdown 不支持的元素
例如，插入具有具体尺寸的图片、视频或音频。

**这里插入一张图片**
<img src="https://github.githubassets.com/images/icons/emoji/octocat.png" alt="Logo" width="50" height="50">
<br>

**这里插入一个网络视频**
<iframe src="//player.bilibili.com/player.html?isOutside=true&aid=19390801&bvid=BV1bW411n7fY&cid=31621681&p=1" scrolling="no" border="0" frameborder="no" framespacing="0" allowfullscreen="true"></iframe>

#### 3.使用更精确的文本语义标签
HTML 提供了比 Markdown 更丰富的文本语义标签。

<abbr title="HyperText Markup Language">HTML</abbr>
<cite>—— 引用来源</cite>
文本<sub>下标</sub>和文本<sup>上标</sup>
<kbd>Ctrl</kbd> + <kbd>C</kbd> 用于复制。


#### 4. 实现复杂布局（简单表格、分栏等）
对于复杂的多列布局或合并单元格的表格，直接使用 HTML 比 Markdown 表格更可靠

<table>
  <tr>
    <th colspan="2">合并的表头</th>
  </tr>
  <tr>
    <td>单元格 A</td>
    <td rowspan="2">跨行的单元格</td>
  </tr>
  <tr>
    <td>单元格 C</td>
  </tr>
</table>

### 注意事项
1. **安全性与兼容性：** 并非所有 Markdown 渲染环境（尤其是某些在线平台）都允许或安全地渲染任意 HTML。在发布前，请确认目标平台的支持情况。

2. **可读性与维护性：** 过度使用 HTML 会丧失 Markdown 简洁易读的优势。仅在必要时使用。

3. **替代方案：** 对于常见需求（如上标/下标），许多现代 Markdown 解析器（如 Pandoc、Typora）提供了扩展语法（如 x^2^、H~2~O），这通常是比 HTML 更优雅的跨平台选择。

### 最佳实践
- **保持简洁**：尽量使用 Markdown 语法，仅在必要情况下使用 HTML。

## 高级功能

这些功能属于扩展语法，并非所有 Markdown 解析器都原生支持，但在许多现代工具和平台（如 GitHub、GitLab、Typora、Obsidian、各种静态网站生成器）中已成为标配。

### 1. 脚注
用于在页面底部添加注释或引用来源。

**语法**： 在需要添加脚注的文本后面加上 `[^footnote]`，然后在文档末尾添加脚注定义。不在文档末尾也行，但脚注的解释会自己放在文档末尾，并且自动添加一个横线。

**示例**：
这是一个带有脚注的文本[^1]。
56号水泥拌蜜雪冰城可以打通地道开私车[^ft]。

[^1]: 这是一个脚注。脚注内容可以包含多行文本。
[^ft]: 私车就是卫龙辣条。


### 2. 目录 (TOC)
这是一个特殊标记。当解析器遇到它时，它会自动扫描当前文档的所有标题（通常是 # 到 ###### 级别的 Markdown 标题），然后在此处生成一个包含层级结构的链接目录。

#### 工作原理
1.**解析标题：**工具会读取文档，找出所有用 # 语法定义的标题。

2.**构建层级：**根据标题的级别（H1, H2, H3 等）创建嵌套的列表结构。

3.**生成锚点链接：**为每个标题创建一个跳转链接（通常是 id 锚点），点击目录项可以直接跳转到文档的相应部分。

**语法**：在文档中添加 `[TOC]` 或者 ，解析器会自动将其替换为生成的目录。

**示例 [TOC]** ：
[TOC]
**示例 [[toc]]** ：
[[toc]]

*注意：这是一个非标准指令，具体语法（如 `[[toc]]`、`@[toc]`）取决于你使用的工具。*

### 3. 数学公式
使用 LaTeX 语法嵌入数学公式，通常需要渲染引擎（如 MathJax 或 KaTeX）支持。

- **行内公式**：用单个美元符号 `$` 包裹。
    质能方程是 $E = mc^2$。

- **块级公式**：用两个美元符号 `$$` 包裹并独立成行。
    $$
    \int_a^b f(x)dx = F(b) - F(a)
    $$

通过特定的代码块语言标识符来绘制图表。最常见的是 **Mermaid**，它被 GitHub、GitLab、Obsidian 等广泛集成。

## 常用工具

工欲善其事，必先利其器。选择合适的工具可以极大提升 Markdown 写作和管理的效率。

### 编辑器
1.  **Visual Studio Code (VS Code)**
    - **平台**：全平台
    - **特点**：强大的代码编辑器，通过插件（如 `Markdown All in One`, `Markdown Preview Enhanced`）成为顶级的 Markdown 环境。支持实时预览、目录生成、格式整理等。

2.  **Typora**
    - **平台**：全平台
    - **特点**：优雅的“所见即所得”编辑器，专注于沉浸式写作。现已转为付费软件，但仍广受好评。

3.  **Obsidian**
    - **平台**：全平台
    - **特点**：以“双向链接”和“知识图谱”为核心的笔记软件，使用本地 Markdown 文件，插件生态极其丰富。

4.  **Notion / 语雀 / 飞书文档**
    - **平台**：Web、移动端
    - **特点**：在线协作文档工具，底层编辑体验借鉴了 Markdown（支持部分语法快捷输入），但最终存储格式非标准 `.md` 文件。

### 转换与发布工具
1.  **Pandoc**
    - **特点**：“文档转换的瑞士军刀”。可将 Markdown 转换为 HTML、PDF、Word、Epub、演示文稿等数十种格式。是自动化出版流程的核心工具。
    - **命令示例**：`pandoc input.md -o output.pdf`

2.  **静态网站生成器 (Static Site Generators, SSG)**
    - **常见工具**：Hugo, Jekyll, Hexo, VuePress, Docusaurus。
    - **特点**：将用 Markdown 编写的文章，配合模板和主题，批量生成为静态 HTML 网站。是创建技术博客、项目文档、个人网站的主流方式。

### 在线平台与协作
1.  **GitHub / GitLab / Gitee**
    - **特点**：代码托管平台。其 `README.md`、`issues`、`wiki` 均原生支持并完美渲染 GFM (GitHub Flavored Markdown)。是程序员展示项目的首要阵地。

2.  **HackMD / StackEdit**
    - **特点**：功能强大的在线 Markdown 编辑器，支持实时协作、发布、导出，是团队协作编写文档的绝佳选择。

### 浏览器插件
1.  **Markdown Viewer**
    - **作用**：在浏览器中直接预览本地 `.md` 文件。
2.  **Markdown Here**
    - **作用**：可以在支持富文本编辑的网页（如 Gmail、微信公众号后台）中，将粘贴的 Markdown 文本一键转换为格式正确的 HTML。

### 选择建议
- **程序员或技术写作者**：首选 **VS Code**（生态强大）或 **Obsidian**（知识管理）。
- **追求简洁写作体验**：选择 **Typora**。
- **团队内容协作**：使用 **Notion**、**语雀** 或 **HackMD**。
- **构建博客或文档站**：学习并使用 **Hugo** 或 **VuePress**。
- **万能格式转换**：掌握 **Pandoc**。

## 最佳实践

遵循良好的写作习惯，可以让你的 Markdown 文档更专业、更易维护，并在不同平台间获得一致的渲染效果。

### 1. 保持格式一致
- **列表符号**：在同一文档中，选择一种无序列表符号（`-`、`*` 或 `+`）并坚持使用。**推荐使用 `-`**，因为它最清晰，且不与强调符号混淆。
- **标题空行**：在标题前后保留空行，这能确保几乎所有解析器都能正确识别标题。
- **链接引用**：对于长文档或技术文档，**使用参考式链接**将 URL 统一放在文档末尾，让正文更清晰，且便于集中管理。

### 2. 提高可读性与兼容性
- **空格使用**：
    - 在 `#` 与标题文字、列表标记与内容之间使用**一个空格**。
    - 在表格的管道符 `|` 和单元格内容之间添加空格，使源码对齐。
- **行宽限制**：建议将每行文本控制在 **80-120 个字符**以内。这不是硬性要求，但能让你在代码编辑器或终端中无需横向滚动即可阅读源码，这对版本控制的差异对比（diff）也更友好。
- **转义歧义字符**：当普通文本中可能被误解为 Markdown 语法的字符（如 `*`、`_`）时，使用反斜杠 `\` 进行转义，或将其放入行内代码中。

### 3. 优化文档结构
- **善用目录**：对于超过一屏的长文档，在开头使用目录（手动或自动生成）能极大提升导航体验。
- **标题层级清晰**：按逻辑顺序使用标题（`H1` -> `H2` -> `H3`），不要跳过层级（例如避免从 `H1` 直接跳到 `H3`）。
- **分割长文档**：如果单个文档过长，考虑将其拆分为多个相互链接的文档。

### 4. 图片与资源管理
- **使用有意义的替代文本**：`![ ]` 中的替代文本不仅为了无障碍访问，也是图片加载失败时的关键信息。
- **优先使用相对路径**：对于项目内的图片或链接，使用相对路径（如 `./images/logo.png`）。这能确保项目在移动或克隆后，链接仍然有效。
- **压缩大图片**：在插入图片前，使用工具对其进行压缩，以减小文档仓库体积并提高加载速度。

### 5. 代码展示
- **指定语言**：在围栏代码块中**始终指定语言**，以获得正确的语法高亮。
- **避免过长的代码块**：如果代码块非常长，考虑将其内容提取到独立的脚本文件中，并在文档中提供文件路径和简要说明。

### 6. 版本控制协同
- **以换行符结束文件**：确保文件末尾有一个空行，这是一个 POSIX 标准，能让许多命令行工具处理得更顺畅。
- **提交信息关联文档**：当修改文档时，在 Git 提交信息中简要说明，如 `docs: 更新安装说明`。
- **审阅渲染效果**：在提交或发布前，使用多个工具（如 GitHub 预览、本地编辑器）检查文档的最终渲染效果，确保格式符合预期。

遵循这些实践，你写出的 Markdown 将不仅是“能渲染”，更是“优雅、健壮且专业”的。

## 扩展语法

“Markdown”在不同语境下可能指代不同的具体实现。除了John Gruber的最初规范，一些重要的变体和扩展标准已成为特定领域的事实标准。

### 1. GitHub Flavored Markdown (GFM)
由GitHub制定，是当今最流行、影响力最大的Markdown扩展集。
- **核心新增**：
    - 表格
    - 任务列表 (`- [ ]`)
    - 自动链接 (自动识别URL和邮箱)
    - 围栏代码块和语法高亮的标准化
    - 删除线 (`~~`)
- **特点**：完全兼容CommonMark，并添加了上述实用功能。在GitHub、GitLab、Gitee及许多现代编辑器中默认支持。

### 2. CommonMark
一个旨在将Markdown语法严格标准化、消除歧义的项目。
- **核心目标**：提供一个**明确、严谨的规范**，让不同解析器的渲染结果完全一致。
- **特点**：它没有新增太多“功能”，而是精确定义了原始Markdown中含糊的部分（如嵌套强调、链接引用）。GFM基于CommonMark构建。

### 3. Markdown Extra (PHP Markdown Extra)
主要用于PHP环境（如早期的一些Wiki和CMS），引入了一些在静态网站生成器中流行的概念。
- **核心新增**：
    - 定义列表
    - 带类/ID属性的Markdown元素
    - 围栏代码块
    - 脚注
    - 表格
- **影响**：其许多功能被后来的工具（如某些Markdown编辑器和静态网站生成器）吸收或借鉴。

### 4. 各平台/工具特有的扩展
- **Obsidian**：
    - 内部链接（双链）：`[[笔记名]]`、`[[笔记名#标题]]`
    - 属性（Properties）：在Front Matter中定义元数据。
    - 查询（Dataview）：通过代码块查询笔记。
- **Typora**：
    - 支持GFM和许多CommonMark特性。
    - 对数学公式、图表、目录等高级功能有良好的内置支持。
- **静态网站生成器（Hugo, Jekelly等）**：
    - 短代码（Shortcodes）：在Markdown中插入复杂的自定义HTML或功能组件，如 `{{< youtube id >}}`。
    - 自定义变量：在模板中引用Front Matter定义的变量。

### 选择与兼容性建议
1.  **默认使用GFM**：对于技术文档和通用写作，以GFM为基准能获得最广泛的兼容性。
2.  **确认目标平台**：在写作前，了解文档最终发布平台（如GitHub、公司Wiki、特定博客系统）支持哪些语法。
3.  **慎用“黑魔法”**：对于Obsidian内部链接、特定SSG的短代码等高度特定于某个工具的功能，需意识到它们在其他平台将无法渲染，甚至可能破坏文档结构。如果文档需要迁移或跨平台发布，应尽量避免使用，或提供备用方案。

理解这些扩展的差异，能帮助你在享受便利功能的同时，写出更具可移植性的文档。

## 总结

### Markdown 的核心价值
Markdown 的成功在于它完美地平衡了**可读性**、**易写性**和**功能性**。它让作者能专注于内容创作本身，而非格式调整，同时产出结构清晰、可直接用于发布的文档。

### 技术定位
- **它不是编程语言**，而是一种轻量级**标记语言**。
- **它不是排版工具**，无法也不应追求像素级精确的版面控制，其核心是定义文档结构。
- **它是转换中枢**：以纯文本为基石，可向 HTML、PDF、Word 等多种格式无缝转换。

### 学习路径建议
1.  **掌握核心**：首先熟练运用标题、列表、链接、图片、代码块、粗体/斜体这六大基础语法。
2.  **熟练工具**：选择一个顺手的编辑器（如 VS Code 或 Typora），并学会使用预览功能。
3.  **了解扩展**：根据你的主要使用场景（如 GitHub 写 README、用 Obsidian 记笔记、写技术博客），学习对应的扩展语法（GFM、双向链接、Front Matter 等）。
4.  **融入流程**：将 Markdown 与版本控制（Git）、文档站点生成（Hugo/VuePress）、自动化发布等工具结合，构建高效的知识管理和技术写作工作流。

### 局限性认知
了解其局限能让你在正确的场景使用它：
- **复杂排版**：不适合需要精密版面设计的传单、海报等。
- **标准化差异**：不同平台渲染细节可能有微小差异。
- **动态内容**：原生不支持嵌入动态脚本或复杂交互。

尽管如此，对于绝大多数以**内容为核心**的写作场景——技术文档、笔记、博客、邮件、报告——Markdown 仍然是当前最优秀、最普适的解决方案之一。
